Code Map — "Where Is X Written?"

Third companion to exam-study-guide.html and qa-guide.html. This file answers questions of the form "where does the code do X?" with exact file:line references, the actual code excerpt, and a one-line explanation of why it lives there.

How the examiner will use this kind of question:
  • They'll point at a behavior ("OK, where do you decide it's safe to fire?") and expect you to navigate to the exact file and line.
  • They might ask "if I wanted to change the laser duration to 5 seconds, what file would I edit?" — that's a config-vs-code-vs-hardware question.
  • They may follow up with "and why is it there and not in this other file?" — that's the one-owner-per-hardware design question.
How to use this guide: read the table of contents, then drill the sections in order. Each entry gives the file:line tag, a code excerpt, and a spoken explanation.

B. Hardware initialisation

Q. Where is the PCA9685 I²C address set?
servo.py:54 
54 PCA9685_ADDRESS: int = 0x40
55 PAN_CHANNEL: int = 0
56 TILT_CHANNEL: int = 1

Why here: servo.py is the one owner of the PCA9685. Any other file changes its mind about the address by importing this module, not by setting its own constant. The default 0x40 means "no A0–A5 jumpers shorted on the board."

Q. Where is the PWM pulse range configured?
servo.py:60–62, 81–84 
60 # DS3225 needs 500-2500 µs pulses over a 270° range. ServoKit defaults to
61 PULSE_MIN_US: int = 500
62 PULSE_MAX_US: int = 2500
63 ACTUATION_RANGE_DEG: int = 270

82     """Apply DS3225-specific pulse range and actuation range to a channel."""
83     kit.servo[channel].set_pulse_width_range(PULSE_MIN_US, PULSE_MAX_US)
84     kit.servo[channel].actuation_range = ACTUATION_RANGE_DEG

Why here: DS3225 isn't a standard hobby servo (180°/1000–2000 µs). If we skipped set_pulse_width_range and actuation_range the library would use defaults and our calibrated 220° pan limit would be unreachable. _configure_channel is called once per servo in init().

Q. Where is the laser pin claimed and forced LOW?
laser.py:76–79 
76     laser_dev = LED(LASER_PIN)
77     laser_dev.off()
78     log.info("Laser initialized on GPIO%d (OFF)", LASER_PIN)
79     return laser_dev

Why here: gpiozero.LED(18) opens /dev/gpiochip0, claims pin 18 as output, and (by default) drives it LOW. We also call .off() explicitly so the intent is in the code, not relying on the library's default — defensive against future gpiozero changes.

C. I²C / PWM / servo control

Q. Where does a Python "set angle" call become an I²C write?
servo.py:189–196 (and servo.py:220–223 for tilt) 
189     if ramp:
190         _pan_current = _ramp(kit, PAN_CHANNEL, _pan_current, clamped)
191     else:
195         kit.servo[PAN_CHANNEL].angle = clamped
196         _pan_current = clamped

That single Python attribute assignment on line 195 is the trigger. ServoKit converts the angle into a pulse width (using our calibrated 500–2500 µs / 270° range), maps it to a 12-bit "OFF" count for the PCA9685, and writes the LED0_ON/OFF or LED1_ON/OFF registers via the adafruit_pca9685 driver. That driver in turn calls into adafruit_blinka → Linux /dev/i2c-1, which puts 5 bytes on the I²C wire at 100 kHz.

Q. Where is the channel-to-servo mapping (pan = channel 0, tilt = channel 1)?
servo.py:55–56 
55 PAN_CHANNEL: int = 0
56 TILT_CHANNEL: int = 1

If we ever swap which servo header the wires go into, only these two constants need updating — the rest of servo.py uses them by name.

Q. Where is the ramping behaviour implemented?
servo.py:87–109 
87 def _ramp(kit: ServoKit, channel: int, current: float, target: float) -> float:
88     ...
102     direction = 1 if target > current else -1
103     angle = current
104     while abs(target - angle) > RAMP_RESOLUTION_DEG:
105         angle += direction * RAMP_RESOLUTION_DEG
106         kit.servo[channel].angle = angle
107         time.sleep(RAMP_DELAY_S)

Steps in 2° / 50 ms increments (constants RAMP_RESOLUTION_DEG = 2.0, RAMP_DELAY_S = 0.05 on lines 66–67). Used by calibration and centering; not used by tracking — tracker.update always calls move_pan / move_tilt with ramp=False to avoid stalling the per-frame loop.

D. Safety clamps & limits

Q. Where are the servo angle clamps enforced? (the safety-critical line)
servo.py:181 (pan) · servo.py:212 (tilt) 
180 
181     clamped = max(PAN_MIN, min(PAN_MAX, angle))
182     if clamped != angle:
183         log.warning(
184             "Pan request %.1f° clamped to %.1f° (limits %.1f°-%.1f°)",
185             angle, clamped, PAN_MIN, PAN_MAX,

This is the single line that prevents the bracket from being driven into a hard stop. Every caller — PID, calibration, manual control, anything — funnels through move_pan/move_tilt, so this clamp is the global safety net. The warning log makes clamping visible during demos.

Q. Where are the actual limit values stored?
servo.py:40–43

Why in servo.py and NOT config.py: these are hardware-safety values, not tuning values. They were measured with calibrate_servo.py against the actual bracket and represent physical reality, not user preference. The centers are computed on lines 47–48, so asymmetric ranges (tilt 115–205, centred at 160 not 135) come out right automatically.

E. Laser control logic

Q. Where does GPIO18 actually go HIGH?
laser.py:82–85 
82 def fire(laser_dev: LED) -> None:
83     """Drive GPIO18 HIGH — laser ON. Caller is responsible for timing."""
84     laser_dev.on()
85     log.info("Laser ON")

One line. laser_dev.on() goes through gpiozero → pin_factory → lgpio → /dev/gpiochip0 → BCM2711 GPSET register → physical pin 12 rises to 3.3 V. The caller (main.py) is responsible for the off-timer; this function does nothing else.

Q. Where is the pin number 18 specified?
laser.py:53 
53 LASER_PIN: int = 18

BCM 18 = physical header pin 12. Comment on lines 46–52 explains why this pin (it's adjacent to a GND pin so the two-wire run is short, and it was an unallocated GPIO in earlier phases).

Q. Where does the laser get forced OFF on exit, even on crash?
main.py:290–297 
290     finally:
291         # Order matters: laser off FIRST (most safety-critical), then
292         # release servos (which will re-center), then camera.
293         log.info("Shutting down...")
294         try:
295             laser.cleanup(laser_dev)
296         except Exception:
297             log.exception("Laser cleanup failed")

The try/finally is the safety guarantee. Whether main() exits cleanly, crashes, or gets Ctrl+C'd, this block runs. Laser is cleaned up BEFORE servos and camera because it's the safety-critical resource.

Q. Where is the cleanup function defined?
laser.py:94–112 
94 def cleanup(laser_dev: LED) -> None:
102     try:
103         laser_dev.off()
104     except Exception:
105         log.exception("Error driving laser pin LOW during cleanup")
106 
107     try:
108         laser_dev.close()
109     except Exception:
110         log.exception("Error closing laser device during cleanup")

Two-step: drive LOW first, then release the GPIO line. Each step is wrapped so a failure in one doesn't prevent the other. Errors are logged but not re-raised — cleanup is called from finally blocks and must not mask the original exception.

G. Detection pipeline

Q. Where is the largest blob picked?
detector.py:86–96 
86     contours, _ = cv2.findContours(
87         mask, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_SIMPLE
88     )
89     if not contours:
90         return None
91 
92     largest = max(contours, key=cv2.contourArea)
93     area = cv2.contourArea(largest)
94     if area < config.MIN_CONTOUR_AREA:
95         return None

Two filters: must exist at all (line 89), must be bigger than 200 px (line 94, config.MIN_CONTOUR_AREA). Below 200 px is noise.

Q. Where is the centroid computed?
detector.py:101–108 
101     moments = cv2.moments(largest)
102     if moments["m00"] == 0:
103         return None
104 
105     cx = int(moments["m10"] / moments["m00"])
106     cy = int(moments["m01"] / moments["m00"])
107     return cx, cy

Image moments give centroid analytically: m00 = area, m10 / m00 = mean x, m01 / m00 = mean y. The m00 == 0 guard prevents divide-by-zero on degenerate contours (shouldn't happen after the area check, but defensive).

H. PID, deadband, coast, recenter

Q. Where is the pixel error computed?
tracker.py:251–259 
251     target_x, target_y = target_pos
258     pan_error = target_x - config.FRAME_CENTER_X
259     tilt_error = target_y - config.FRAME_CENTER_Y

This is where camera coordinates become "how far from centre." FRAME_CENTER_X / Y are 320 and 240, defined in config.py:22–23. Positive pan_error = target right of centre. Positive tilt_error = target BELOW centre (OpenCV y increases downward).

Q. Where does the PID actually run?
tracker.py:263–264 
263     pan_correction = pan_pid(pan_error)
264     tilt_correction = tilt_pid(tilt_error)

Calling the PID object as a function feeds it the current error and returns a correction (degrees, already clamped by the controller's output_limits). simple_pid manages its own internal time-stepping for the I and D terms.

Q. Where is the deadband checked? (this is what gates the "LOCKED" state)
tracker.py:273–278

275     in_deadband = (
276         abs(pan_error) < deadband_px
277         and abs(tilt_error) < deadband_px

Both axes must be within 15 px (the default deadband). The result is returned to main.py as result["in_deadband"], which is what the state machine reads to know "target locked." Same value gates fire permission.

When the detector returns None, we coast if (a) we still have coast budget left and (b) the last correction was non-trivial (target wasn't sitting still in the deadband when it disappeared). The coast budget is reset to COAST_MAX_FRAMES = 30 every successful tracking step (line 312).

I. State machine & fire trigger

Q. Where are the state names defined?
main.py:86–89 
86 S_DISARMED = "DISARMED"
87 S_ARMED    = "ARMED"
88 S_FIRING   = "FIRING"
89 S_COOLDOWN = "COOLDOWN"

Plain strings — no enum needed for four states. Used as the state variable on line 216 (state = S_DISARMED) and compared throughout the loop.

Q. Where does the loop transition FIRING → COOLDOWN?
main.py:230–237 
230     if state == S_FIRING and now - t_fire_start >= FIRE_DURATION_S:
231         laser.off(laser_dev)
232         cooldown_until = now + COOLDOWN_S
233         state = S_COOLDOWN
234         log.info("Fire complete — cooling down for %.1fs.", COOLDOWN_S)
235     elif state == S_COOLDOWN and now >= cooldown_until:
236         state = S_ARMED

Time-based, checked at the top of every frame. The fire ON event itself happens elsewhere (the F keypress handler); this block handles the auto-OFF and auto-arm-recovery.

Q. Where is the F key handled?
main.py:272–281 
272     elif key == ord('f'):
273         if state != S_ARMED:
274             log.warning("Fire ignored — state is %s (must be ARMED).", state)
275         elif not locked:
276             log.warning("Fire ignored — target NOT LOCKED.")
277         else:

Three safety checks before the laser turns on: (1) state must be ARMED — DISARMED can't fire, neither can FIRING/COOLDOWN. (2) locked must be true — set on line 248 from result["in_deadband"]. (3) Implicitly: the key event itself was caught — the user pressed F.

Q. Where is the gpiozero pin_factory closed before forking?
control_panel.py:594–599 (in _on_run_demo; similar block in _on_boresight at 511–515) 
593         # fresh pin_factory automatically.
594         try:
595             from gpiozero import Device
596             if Device.pin_factory is not None:
597                 Device.pin_factory.close()
598                 Device.pin_factory = None
599                 log.info("Closed gpiozero pin_factory to fully release chip handle.")

This is the fix for the "GPIO busy" subprocess bug. Closing the singleton pin_factory releases the chip-level FD; setting it to None forces gpiozero to lazily recreate on the next LED() call. The 0.3 s sleep that follows lets kernel-level state settle before Popen forks.
