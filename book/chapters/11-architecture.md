# 11. תיאור הארכיטקטורה

הפרק מציג את ארכיטקטורת המערכת בגישת **Top-Down Level Design** —
מהתיאור הגבוה ביותר של המערכת כיחידה אחת, ועד פירוט הפינים והקבצים.
הארכיטקטורה מחולקת לשני "מסלולים" שמתוארים במקביל: **חומרה**
(הרכיבים הפיזיים והחיבורים ביניהם) ו**תוכנה** (המודולים והקריאות
ביניהם). הפרק נסגר בתיאור הזרימה בין השכבות בלולאת הריצה.

---

## 11.1 גישת התיאור

Top-Down Level Design הוא תהליך תיעוד שעובד **מהכללי לפרטי**:

- **Level 0** — המערכת כיחידה אחת. קלטים, פלטים, תפקיד כללי.
- **Level 1** — חלוקת המערכת לתת־מערכות פונקציונליות.
- **Level 2** — הרכיבים בתוך כל תת־מערכת.
- **Level 3** — הפונקציות/הפינים/הקבצים בתוך כל רכיב.

היתרון של הגישה: כל קורא יכול לעצור ברמת הפירוט שמתאימה לו. מנהל
פרויקט קורא רק את Level 0–1; טכנאי שעובד על המעגל זקוק ל־Level 3.

---

## 11.2 Level 0 — המערכת כיחידה אחת

ברמת ההפשטה הגבוהה ביותר, המערכת היא **תת־מערכת אחת** עם זרמי
קלט ופלט:

![תרשים מערכת ברמה 0](system-overview.png)

**קלטים:**
- 12V מספק הקיר.
- 5V דרך USB-C ל־Pi.
- זרם וידאו ממצלמת USB (~30 MB/s).
- אישור מפעיל (`'f'` במקלדת או כפתור ב־GUI).

**פלטים:**
- תנועת פאן (170° טווח).
- תנועת טילט (90° טווח).
- אלומת לייזר (5mW, 650nm).
- לוג טקסט + overlay חזותי על חלון OpenCV.

**הפעולה הכוללת:** המערכת קולטת תמונה, מאתרת מטרה צבעונית בה,
מחשבת סטיית פיקסלים ממרכז הפריים, ממירה אותה לתיקון זוויתי, שולחת
פקודות לסרוו, ומפעילה לייזר על אישור — כל זאת בלולאה רציפה
ב־30 fps.

## 11.3 Level 1 — תת־מערכות

המערכת מורכבת מ־**שש תת־מערכות פונקציונליות**:

| # | תת־מערכת | תפקיד | רכיבים עיקריים |
|---|---|---|---|
| 1 | **ספק כוח** (Power) | המרה ופיצול של החשמל לכל הרכיבים | ספק 12V, LM2596, ספק USB-C |
| 2 | **מחשוב** (Compute) | עיבוד, החלטה, בקרה | Raspberry Pi 4B + Raspberry Pi OS |
| 3 | **חישה** (Sensing) | קליטת תמונה מהעולם | LifeCam HD-3000 |
| 4 | **הפעלה** (Actuation) | תנועה מכאנית של מצלמה + לייזר | PCA9685 + 2× DS3225 + ערכת פאן־טילט |
| 5 | **פליטה** (Emission) | פליטת אלומת לייזר | MOSFET + נגדים + דיודת לייזר |
| 6 | **ממשק משתמש** (UI) | קלט מהמפעיל + תצוגת סטטוס | tkinter GUI + OpenCV overlay + מקלדת |

האינטראקציות בין תת־המערכות:

```
[ספק כוח] ──→ [מחשוב, הפעלה, פליטה]
[חישה] ──→ [מחשוב]
[מחשוב] ──→ [הפעלה, פליטה]
[ממשק משתמש] ←→ [מחשוב]
```

### 11.5.1 רמה 3 — חיווט פיזי

הפרטים הנמוכים ביותר של ארכיטקטורת החומרה — איזה פין מתחבר לאן.
ראו [§15.2 — חיבורים ואופן הפעולה](#) לטבלת חיווט מלאה. סיכום
תמציתי:

| מקור | יעד | פין/חוט | תפקיד |
|---|---|---|---|
| Pi GPIO Pin 2 | PCA9685 VCC | חוט אדום | אספקת לוגיקה 5V ל־PCA9685 |
| Pi GPIO Pin 3 (SDA) | PCA9685 SDA | חוט סגול | נתוני I²C |
| Pi GPIO Pin 5 (SCL) | PCA9685 SCL | חוט אפור | שעון I²C |
| Pi GPIO Pin 6 (GND) | Common GND | חוט שחור | אדמה משותפת |
| Pi GPIO Pin 4 (5V) | מסילת לייזר | חוט אדום | אספקה ללייזר דרך 100Ω |
| Pi GPIO Pin 12 (GPIO18) | MOSFET Gate | חוט צהוב | פקודת לייזר (3.3V) דרך 220Ω |
| Pi GPIO Pin 14 (GND) | Common GND | חוט שחור | אדמת מסילת הלייזר |
| ספק 12V | LM2596 IN | חוט אדום עבה | מתח כניסה |
| LM2596 OUT+ | PCA9685 V+ (גרין טרמינל) | חוט אדום | אספקת כוח לסרוו (5V/3A) |
| LM2596 OUT− | Common GND | חוט שחור | אדמה משותפת |
| PCA9685 ערוץ 0 | סרוו פאן (חוט כתום) | — | פקודת PWM פאן |
| PCA9685 ערוץ 1 | סרוו טילט (חוט כתום) | — | פקודת PWM טילט |

### 11.5.2 מסילות החשמל

המערכת מתחזקת **שתי מסילות חשמל נפרדות** עם GND משותף:

- **מסילת לוגיקה (5V Logic):** מ־Pi GPIO Pin 2 → PCA9685 VCC.
  צריכה נמוכה (~50 mA), רגישה לרעש (אותות I²C).
- **מסילת סרוו (5V Servo):** מ־LM2596 OUT+ → PCA9685 V+.
  צריכה גבוהה (פיקים של 2A), פחות רגישה לרעש.

ההפרדה הזו פותרת את הבעיה שתועדה ב־[`problems/001-servo-power.md`](../problems/001-servo-power.md):
פיקי זרם של הסרוו לא משפיעים על מסילת הלוגיקה.

## 11.6 ארכיטקטורת תוכנה — Top-Down

תרשים יחסים בין המודולים:

![תרשים תוכנה Top-Down](software-topdown.png)

### 11.6.1 רמה 1 — שכבות

הקוד מאורגן בארבע שכבות מובחנות:

| שכבה | אחריות | קבצים |
|---|---|---|
| **Orchestration** | בקרת זרימה ראשית, ממשק משתמש, מחזור חיים | `main.py`, `control_panel.py`, `test_tracking.py` |
| **Logic** | אלגוריתמים: זיהוי, בקרה | `detector.py`, `tracker.py` |
| **Hardware Abstraction** (Owner Modules) | תרגום בין לוגיקה לפעולות חומרה | `servo.py`, `camera.py`, `laser.py` |
| **Configuration** | מקור־אמת לקבועים מכוילים | `config.py` |

כל שכבה מותרת לקרוא רק לשכבות מתחתיה — לעולם לא הפוך. זה מבטיח
שאין תלויות מעגליות.

### 11.6.2 רמה 2 — מודולי בעלות בלעדית (Owner Modules)

ליבת עיצוב התוכנה. **כל מודול אחראי על תת־מערכת חומרה אחת**,
ואף מודול אחר אינו רשאי לייבא את ספריית הצד־השלישי השייכת לאותה
תת־מערכת:

| מודול | תת־מערכת חומרה | ספריות צד־שלישי שאסור לאחרים לייבא |
|---|---|---|
| `servo.py` | PCA9685 + DS3225 | `adafruit_servokit`, `adafruit_pca9685` |
| `camera.py` | LifeCam HD-3000 | (`cv2.VideoCapture` — רק כאן) |
| `detector.py` | אלגוריתם זיהוי | (`cv2` לעיבוד תמונה — בסדר כאן) |
| `tracker.py` | בקרי PID | `simple_pid` |
| `laser.py` | GPIO18 + דיודת לייזר | `gpiozero` (לפין הזה) |
| `config.py` | כל הקבועים המכוילים | — |
| `control_panel.py` | tkinter GUI | `tkinter` |

בפועל: אם `test_tracking.py` רוצה להזיז סרוו, **חובה** עליו לקרוא
ל־`servo.move_pan(kit, 150.0)` ולא לכתוב `kit.servo[0].angle = 150.0`
ישירות, גם כשזה פשוט יותר. הסיבה — ה־clamping לטווח [50°, 220°]
מתבצע בתוך `servo.py`. עקיפת המודול מסכנת את הסרוו.

### 11.6.3 רמה 3 — פונקציות פנימיות לכל מודול

כל מודול חושף API ציבורי קטן ושומר את הפנימיות פרטיות (קונבנציית
`_underscore_prefix` של Python). דוגמה מ־`servo.py`:

- API ציבורי: `init()`, `move_pan()`, `move_tilt()`, `center()`,
  `cleanup()`, `current_pan()`, `current_tilt()`.
- פנימי: `_configure_channel()`, `_ramp()`, ומשתני מצב
  `_pan_current`, `_tilt_current`.

פירוט מלא של ה־API לכל מודול ב־§13 (ספריות).

### 11.6.4 גרף תלויות בקוד

```
main.py / control_panel.py  →  servo, camera, detector, tracker, laser, config
tracker.py                  →  servo, config (+ simple_pid)
detector.py                 →  config (+ cv2, numpy)
camera.py                   →  (cv2)
servo.py                    →  config (+ adafruit_servokit)
laser.py                    →  (gpiozero)
config.py                   →  (numpy)
```

**אין מעגלים.** `config.py` הוא העלים של העץ — לא מייבא דבר משאר
הפרויקט. כל מודול אחר עשוי לייבא ממנו, אבל אף אחד לא ייבא ממודולים
"מעליו" בהיררכיה.

## 11.7 זרימת נתונים בין השכבות

בכל איטרציה של לולאת ה־tracking, נתונים זורמים ישר ולמעלה:

```
[חיישן]  →  [Hardware Abstraction]  →  [Logic]  →  [Hardware Abstraction]  →  [אקטואטור]

LifeCam  →  camera.capture_frame  →  detector.detect  →  tracker.update  →  servo.move_*  →  PCA9685 → DS3225
                                          (cv2)             (simple_pid)
```

**הטרנספורמציות במידע:**

| שלב | קלט | פלט | זמן (טיפוסי) |
|---|---|---|---|
| `camera.capture_frame()` | (אין; קריאת syscall) | `np.ndarray` 480×640×3 BGR | ~15 ms |
| `detector.detect(frame)` | פריים BGR | `(cx, cy)` או `None` | ~5–8 ms |
| `tracker.update(target_pos)` | `(cx, cy)` | dict עם `pan_correction`, `tilt_correction`, מצב | ~0.1 ms |
| `servo.move_pan(kit, angle, ramp=False)` | זווית במעלות | פקודת I²C נשלחת | ~2 ms |
| `cv2.imshow + waitKey` | פריים עם overlay | תצוגה למסך | ~3 ms |

**סה"כ זמן לפר־פריים: ~26 ms.** בתוך תקציב 33 ms (30 fps). הפנאי
~7 ms זמין לתוספות עתידיות.

## 11.8 זרימת בקרה — מכונת המצבים של הלולאה

בכל איטרציה, `tracker.update()` מחליט באיזה **מצב** המערכת נמצאת
ופועל בהתאם:

```
                    ┌─────────────────────────────────┐
                    ↓                                 │
                Tracking (PID active)  ←─ target re-acquired
                    │                                 │
   error within deadband?         target lost?        │
            yes ↓                       ↓ yes         │
                                                      │
            Locked (no movement)   Coasting (apply    │
                                    last correction   │
                                    × decay, max 30   │
                                    frames)           │
                                       │              │
                                       │              │
                              coast exhausted OR      │
                              both axes clamped       │
                                       ↓              │
                                  Recentering         │
                                  (return to PAN_CENTER, │
                                   TILT_CENTER step   │
                                   by step)           │
                                       │              │
                                  reached center      │
                                       ↓              │
                                  Tracking ──────────┘

           target lost while in deadband (no direction)
                                ↓
                            Holding
```

חמשת המצבים האפשריים:

| מצב | טריגר | פעולה |
|---|---|---|
| **Tracking** | מטרה נמצאת, שגיאה > deadband | PID פעיל; שליחת `pan_correction`, `tilt_correction` לסרוו |
| **Locked** | מטרה נמצאת, שגיאה ≤ deadband | שמירה על מצב; מותר ירייה |
| **Coasting** | מטרה אבדה, התיקון הקודם משמעותי | המשך תיקון אחרון עם decay 0.95/פריים |
| **Recentering** | Coast הסתיים בלי הצלחה, או שני צירים clamped | חזרה למרכז ב־2°/פריים |
| **Holding** | אין מידע (אבדה בעוד ב־deadband) | המתנה ללא תזוזה |

## 11.10 איך הארכיטקטורה תומכת בשדרוגים

החלטות הארכיטקטורה לא נעשו לטובת הקוד הנוכחי לבד אלא גם לטובת
שדרוגים עתידיים (§16). דוגמאות:

- **החלפת אלגוריתם זיהוי** (לדוגמה: HSV → YOLO) דורשת שינוי רק
  ב־`detector.py`. ה־API שלו (`detect(frame) → (x, y) | None`) לא
  משתנה — אף קובץ אחר לא יודע על השינוי.
- **החלפת מצלמה** (לדוגמה: LifeCam → RealSense עם עומק) דורשת שינוי
  רק ב־`camera.py`. אם ה־API נשמר, שאר המערכת לא יודעת.
- **הוספת סרוו שלישי** (לדוגמה: זרוע מכאנית) דורשת ערוץ 2 ב־PCA9685
  + פונקציה חדשה ב־`servo.py`. אין צורך לגעת ב־`tracker.py` אם הסרוו
  השלישי לא נכלל בלולאת PID.
- **החלפת dishboard tkinter ל־Flask + browser** דורשת רק קובץ חדש;
  לא נוגעים בקבצי הלוגיקה והחומרה.

ההפרדה לשכבות + Owner Modules היא **השקעה לעתיד**, לא רק לקוד
הנוכחי.

## 11.11 סיכום

המערכת מתועדת ב־**4 רמות פירוט** (Top-Down):

| רמה | תיאור | מי קורא |
|---|---|---|
| 0 | המערכת כיחידה אחת — קלטים/פלטים | מנהל פרויקט, מנחה |
| 1 | 6 תת־מערכות פונקציונליות | מהנדס מערכת |
| 2 | רכיבים פרטניים בכל תת־מערכת | טכנאי / רוכש |
| 3 | פינים, חוטים, פונקציות | מבצע ההרכבה / מתחזק קוד |

ה־**hardware** מאורגן ב־6 תת־מערכות עם 2 מסילות חשמל נפרדות
שחולקות GND משותף. ה־**software** מאורגן ב־4 שכבות עם 6 Owner
Modules ו־1 קובץ Config — אין תלויות מעגליות, כל שכבה תלויה רק
בשכבות מתחתיה. **זרימת הנתונים** היא חד־כיוונית מהחיישן לאקטואטור,
ו**זרימת הבקרה** מנוהלת ע"י מכונת מצבים בעלת חמישה מצבים שמטפלת
בכל המקרים האפשריים.

הפירוט הטכני של חיווט החומרה ב־§15.1–15.2; הפירוט הטכני של ה־API
של כל מודול תוכנה ב־§13.
