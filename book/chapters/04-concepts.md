# 4. מושגים

הפרק מגדיר את המושגים הספציפיים לפרויקט שדרושים להבנת שאר הפרק.
לפי הנחיית המחוון — לא מוגדרים כאן מושגים כלליים במדעי המחשב
(משתנה, פונקציה, קובץ, מחלקה וכד'), אלא רק מושגים שהקורא שאינו
מומחה לתחום של ראייה ממוחשבת + בקרת היגב + חומרת Pi עלול לפרש לא
נכון. המושגים מסודרים לפי תחום: חומרה ואלקטרוניקה, ראייה ממוחשבת,
בקרה, ארכיטקטורת תוכנה, ומערכת הפעלה.

---

## 4.1 חומרה ואלקטרוניקה

| מושג | הגדרה בהקשר הפרויקט |
|---|---|
| **GPIO** (General-Purpose Input/Output) | פין דיגיטלי על ה־Raspberry Pi שניתן להגדיר כקלט או פלט בתוכנה. עובד ב־3.3V לוגי, מסוגל לזרם של עד ~16mA. בפרויקט: GPIO18 מפעיל את ה־MOSFET של הלייזר. |
| **I²C** (Inter-Integrated Circuit) | פרוטוקול תקשורת סדרתי דו־חוטי (SDA + SCL) שפותח ב־Philips. ה־Pi הוא Master שמדבר עם PCA9685 ככתובת Slave 0x40 בקצב 100 kbps. כל ההתקנים חולקים את אותם שני חוטים. |
| **PWM** (Pulse-Width Modulation) | אות דיגיטלי שמדמה ערך אנלוגי ע"י שינוי היחס בין זמן ON לזמן OFF בכל מחזור (Duty Cycle). בסרוו: התדר קבוע 50Hz (תקופה 20ms), משך הפולס בתוך התקופה (500–2500µs) מקודד את הזווית. |
| **PCA9685** | שבב מ־NXP המייצר 16 ערוצי PWM נפרדים ברזולוציה של 12-bit. מקבל פקודות מה־Pi דרך I²C ומייצר את אותות ה־PWM ברציפות גם כש־CPU עסוק. |
| **סרוו דיגיטלי DS3225** | מנוע מסוג Servo עם בקר פנימי. מקבל פולסי PWM ב־50Hz ומחזיק את הציר בזווית הנדרשת. בפרויקט: 2 יחידות, ערוצי PCA9685 0 (פאן) ו־1 (טילט). אין משוב מיקום — הקוד עוקב אחר הזווית האחרונה שנשלחה. |
| **Pan / Tilt** | שתי דרגות החופש של המערכת. **Pan** = סיבוב סביב ציר אנכי (ימינה/שמאלה). **Tilt** = סיבוב סביב ציר אופקי (למעלה/למטה). יחד מכסות סקטור של כדור. |
| **Stall Current** | הזרם שמושך מנוע סרוו כשהוא חוסם — מנסה לזוז אבל לא יכול. DS3225 מגיע ל־2A+ בסטיל. הסיבה שדרושה אספקה חזקה (LM2596 5A) ולא MB-102 (700mA). |
| **MOSFET** (Metal-Oxide-Semiconductor Field-Effect Transistor) | טרנזיסטור־שדה. הזרם בין Drain ל־Source נשלט על־ידי מתח על ה־Gate. בפרויקט: IRLZ44N (N-Channel, Logic Level — נפתח ב־3.3V Gate). משמש כמתג בין +5V לבין הלייזר. |
| **Low-Side Switching** | טופולוגיית מיתוג שבה המתג (MOSFET) ממוקם בין הלעומס לבין ה־GND, כך שה־Source של ה־MOSFET מחובר ישירות ל־GND והייחוס של מתח ה־Gate יציב. הגישה הפשוטה והבטוחה בפרויקט הזה. |
| **Pulldown Resistor** | נגד (100kΩ בפרויקט) בין פין לוגי לבין GND. תפקידו לשמור על הפין במצב LOW כאשר אף מקור פעיל לא מחובר אליו. הכרחי על Gate של MOSFET בעת ה־boot של ה־Pi כשהפין צף (floating). בלעדיו — סכנת הפעלת לייזר לא־מבוקרת. |
| **Buck Converter** | ממיר DC-DC שמוריד מתח (Step-Down) ביעילות גבוהה (~85%). הפנימי מפעיל ומכבה את מתח הכניסה בתדר גבוה (~150 kHz); סליל וקבל מחליקים. בפרויקט: LM2596 ממיר 12V → 5.0V עם זרם של עד 3A. |
| **CSI** (Camera Serial Interface) | מחבר ייעודי לחיבור מצלמת Raspberry Pi דרך פס ribbon. Pi 4 משתמש במחבר 15-pin, Pi 5 ב־22-pin — לכן מצלמת ה־Pi 5 שהייתה ברשותנו לא תאמה. |
| **UVC** (USB Video Class) | תקן Microsoft/USB־IF לסטרימינג וידאו דיגיטלי דרך USB. הקרנל של Linux כולל את `uvcvideo` כברירת מחדל ולכן כל מצלמת USB סטנדרטית עובדת ללא התקנה. בפרויקט: LifeCam HD-3000. |

## 4.2 ראייה ממוחשבת

| מושג | הגדרה בהקשר הפרויקט |
|---|---|
| **BGR** | סדר ערוצי הצבע ב־OpenCV — Blue, Green, Red. שונה מ־RGB הרגיל; חשוב להכיר אם מציגים תמונה בכלי חיצוני. |
| **HSV** (Hue, Saturation, Value) | מרחב צבע שמפריד בין הגוון לבהירות. **H** = הצבע עצמו על גלגל (ב־OpenCV 0–179), **S** = רוויה (0 אפור → 255 צבע מלא), **V** = בהירות (0 שחור → 255 מלא). יציב לשינויי תאורה הרבה יותר מ־RGB. |
| **Thresholding** (סף) | פעולה שמסווגת כל פיקסל ל"כן/לא" לפי האם ערכיו בטווח נתון. `cv2.inRange(hsv, low, high)` יוצרת מסכה בינארית של פיקסלים שעוברים את הסף. |
| **Morphological Opening** | רצף של פעולה מורפולוגית: `erode` ואז `dilate`. ה־erode מסיר ספקלי רעש בודדים, ה־dilate משחזר את גודל המטרה. שימושי לניקוי מסכה בינארית רועשת. |
| **Contour** | רצף נקודות שמתאר את גבול אזור מחובר במסכה בינארית. `cv2.findContours` מחלץ את כל הקונטורים בתמונה. בפרויקט: בוחרים את הקונטור הגדול ביותר ומחשבים את מרכז המסה שלו. |
| **Image Moments** | סטטיסטיקות שמתארות הת התפלגות הפיקסלים בקונטור. ה־moment $m_{00}$ הוא השטח, $m_{10}/m_{00}$ ו־$m_{01}/m_{00}$ נותנים את מרכז המסה. כל החישוב מבוצע ע"י `cv2.moments`. |
| **Centroid** | מרכז המסה של אזור בתמונה. בפרויקט: הקואורדינטה (cx, cy) שמייצגת את "המקום של המטרה" בפריים. |

## 4.3 בקרה

| מושג | הגדרה בהקשר הפרויקט |
|---|---|
| **Closed-Loop Control** (בקרה סגורה) | בקר שמודד את התוצאה של הפעולה שלו ומשתמש במידע הזה לקבוע את הפעולה הבאה. בפרויקט: הסרוו זז, המצלמה מצלמת את המקום החדש של המטרה, ה־PID מחשב תיקון מחדש. |
| **PID Controller** | משפחת בקרים שמחשבת תיקון פרופורציוני לשגיאה (P), לסכומה לאורך זמן (I), ולקצב השינוי שלה (D): $u(t) = K_p e + K_i \int e\, dt + K_d \dot{e}$. בפרויקט: בקר P בלבד ($K_p = 0.017$, $K_i = K_d = 0$). |
| **Setpoint** | הערך הרצוי של הבקר. בפרויקט: 0 — רוצים ששגיאת הפיקסלים (target_x − center_x) תהיה אפס. |
| **Error** | ההפרש בין הערך הרצוי לבין הערך הנמדד. בפרויקט: סטיית המטרה ממרכז הפריים בפיקסלים. |
| **Output Limits / Clamping** | חיתוך תכנותי של פלט הבקר לטווח בטוח. בפרויקט: `output_limits = (−10°, +10°)` — לא יותר מ־10° תיקון בפריים יחיד, גם אם השגיאה ענקית. מונע קפיצות מסוכנות וחוסר יציבות. |
| **Deadband** (תחום מתים) | טווח של שגיאה קטנה שבה הבקר לא פועל. בפרויקט: 15 פיקסלים בכל ציר. אם המטרה במרכז בקירוב, הסרוו לא זז — מונע רטט עקב רעש הגלאי (~10 פיקסלים פריים־לפריים). |
| **Coast Mode** *(מושג שהוגדר בפרויקט)* | מצב שמופעל כאשר הזיהוי אבד אך התיקון הקודם היה משמעותי. הברקט ממשיך להחיל את התיקון האחרון עם דעיכה של 5% לפריים, עד 30 פריימים (~1 שנייה). מדמה אינרציה — נותן למטרה זמן לחזור לפריים. |
| **Recenter Mode** *(מושג שהוגדר בפרויקט)* | מצב שמופעל כאשר Coast הסתיים בלי שהמטרה חזרה, או כאשר שני הצירים הגיעו לגבול בו־זמנית. הברקט חוזר חלק למרכז המכויל בקצב 2°/פריים. |
| **Locked / Tracking** | שמות סטטוס באוברליי המעקב. **Tracking** = PID פעיל, **Locked** = שגיאה ≤ Deadband (יעד יציב במרכז) — תנאי הכרחי לאישור ירייה. |
| **Boresight** | היסט פיזי בין הציר האופטי של המצלמה לבין ציר הלייזר. גם אם שניהם מקובעים על אותה לוחית טילט, אינם בדיוק על אותו קו. הפרויקט מודד את ההיסט פעם אחת (`boresight.py`) ומוסיף אותו לתיקון ה־PID כקבועים `BORESIGHT_X_OFFSET`, `BORESIGHT_Y_OFFSET`. |

## 4.4 ארכיטקטורת תוכנה

| מושג | הגדרה בהקשר הפרויקט |
|---|---|
| **Owner Module** *(קונבנציה שהוגדרה בפרויקט)* | קובץ Python בודד שמחזיק בלעדיות על תת־מערכת חומרה אחת. אף קובץ אחר אינו רשאי לייבא את ספריית הצד־השלישי השייכת לאותה חומרה. לדוגמה: רק `servo.py` מייבא `adafruit_servokit`. |
| **SSOT** (Single Source of Truth) | עיקרון שלכל ערך מכויל קיים מקום מקורי יחיד במאגר. בפרויקט: `config.py` מחזיק את כל הקבועים (HSV, PID, גבולות סרוו, Deadband). אם משנים HSV — שורה אחת ב־`config.py`, וכל המודולים רואים את השינוי. |
| **SRP** (Single Responsibility Principle) | עיקרון שלכל מודול תחום אחריות אחד צר. בפרויקט: `servo.py` לא יודע על OpenCV, `detector.py` לא יודע על הסרוו. |
| **Defensive Programming** | סגנון כתיבה שמניח שדברים יכולים להשתבש ומכין הגנות. בפרויקט: clamping זוויות, Pulldown על GPIO, `try/finally` בכל סקריפט שמשתמש בחומרה. |
| **Building Block (BB)** | רכיב תוכנה חיצוני שלא נכתב על־ידי הסטודנט אלא שולב כתלות. הבחנה זו נדרשת ב־§13 — שם מובחנים מודולי הפרויקט (`servo.py`, `detector.py`, וכו') מספריות צד־שלישי (OpenCV, simple-pid, ServoKit). |
| **Try / Finally** | מבנה Python שמבטיח שקטע קוד יבוצע גם אם נזרקה חריגה. בפרויקט: בלוק `finally` מבצע `servo.cleanup`, `laser.cleanup`, `camera.release` — מבטיח שהחומרה תחזור למצב בטוח גם בעת תקלה או Ctrl+C. |

## 4.5 מערכת הפעלה ותהליך פיתוח

| מושג | הגדרה בהקשר הפרויקט |
|---|---|
| **Raspberry Pi OS Bookworm** | מערכת ההפעלה של ה־Pi — וריאנט של Debian 12. הפרויקט משתמש בגרסת 64-bit עם kernel 6.x. |
| **venv** (Virtual Environment) | סביבת Python מבודדת שמאפשרת להתקין ספריות לפרויקט בלי לזהם את ה־Python הגלובלי של המערכת. בפרויקט: `~/pi/venv/`. |
| **`--system-site-packages`** | דגל ש־`venv` נוצר איתו, שמאפשר לסביבה הוירטואלית "לראות" גם חבילות apt של מערכת ההפעלה (OpenCV, NumPy, gpiozero). חוסך קומפילציה של ספריות כבדות בתוך כל venv. |
| **SSH** (Secure Shell) | פרוטוקול חיבור מאובטח לטרמינל מרוחק. בפרויקט: גישה ל־Pi מהלפטופ עם `ssh adam@LaserPi.local`. |
| **VNC** (Virtual Network Computing) | פרוטוקול לשיתוף שולחן עבודה מרחוק. בפרויקט: מאפשר לפתוח חלונות OpenCV (`test_tracking.py`, `tune_detector.py`) מהלפטופ. |
| **cron** | מתזמן משימות של Linux. בפרויקט: job ב־Pi שמבצע `git pull` כל דקה, כדי שקוד חדש שמועלה ל־GitHub מהלפטופ יגיע ל־Pi אוטומטית. |
