לאחר שב־§7 הוצגו 15 ההכרעות עם החלופות לכל אחת, הפרק הזה מציג את

## 8.1 הסטאק הנבחר — מבט־על

המערכת בנויה מארבע שכבות שמתחלקות לתפקידים מובחנים:

| שכבה | רכיבים |
|---|---|
| **חומרה — חישה** | LifeCam HD-3000 (USB UVC) |
| **חומרה — מחשוב** | Raspberry Pi 4B 8GB, Raspberry Pi OS Bookworm 64-bit |
| **חומרה — הפעלה** | PCA9685 (I²C) → 2× DS3225 (Pan/Tilt) → ערכת פאן־טילט מתכת + תושבת מצלמה מודפסת 3D |
| **חומרה — פליטה** | IRLZ44N MOSFET + 220Ω/100kΩ/100Ω → דיודת לייזר 5mW 650nm |
| **חומרה — כוח** | ספק 12V/5A → LM2596 (5.0V/3A) למסילת סרוו; Pi GPIO 5V למסילת לוגיקה |
| **תוכנה — בקרת חומרה** | `gpiozero`, `adafruit-circuitpython-servokit`, `adafruit-blinka`, `cv2.VideoCapture` |
| **תוכנה — עיבוד** | `cv2` (HSV, morphology, contours, moments), `numpy` |
| **תוכנה — בקרה** | `simple-pid` |
| **תוכנה — UI** | `tkinter` (control panel) |
| **תוכנה — שפה** | Python 3.11 |
| **אלגוריתמים** | HSV thresholding + P-only PID + Coast/Recenter state machine |

## 8.2 למה הרכיבים האלה — מבט חומרתי

הבחירות החומרתיות לא עומדות לבד; הן רותמות אחת את השנייה.

### 8.2.1 Raspberry Pi 4B 8GB + Bookworm 64-bit

**הליבה.** הבחירה מכתיבה רוב ההמשך:

- מספק מספיק כוח חישוב לעיבוד וידאו ב־30 fps בלי TPU חיצוני.
- ה־OS (Bookworm 64-bit) כולל `python3-opencv` בגרסה מקומפלת ל־ARM64
  ב־apt — חוסך שעות של קומפילציה ידנית.
- USB-A מובנה (LifeCam) + I²C מובנה (PCA9685) + GPIO מובנה (לייזר)
  — אין צורך ב־HAT מיוחד.
- 8GB RAM מאפשרים לרוץ GUI + tracking + logging במקביל בלי swap.

### 8.2.2 PCA9685 + DS3225 (זוג טבעי)

הזוג הזה הוא הסטנדרט בקהילת ה־Pi:

- PCA9685 מנתק את ייצור ה־PWM מה־CPU — אין jitter גם כש־CPU עסוק
  בעיבוד תמונה.
- ל־DS3225 דרישות PWM זהות לסטנדרט (50Hz, 500–2500µs) — תאמת מלאה.
- ServoKit Adafruit עוטף בדיוק את הצירוף הזה ב־API של שורה אחת.
- שני ערוצי PCA9685 (0 = פאן, 1 = טילט) — מנצלים פחות מ־13% מ־16
  הערוצים, כלומר יש מרחב להוספת סרוו נוסף (זרוע / זום) בעתיד בלי
  להחליף שבב.

### 8.2.3 LM2596 + שתי מסילות נפרדות (פתרון ל־problem 001)

ההפרדה בין מסילת הלוגיקה למסילת הסרוו היא **הבחירה הקריטית בפועל**:

- LM2596 (3A) מטפל בפיקי הזרם של DS3225 (2A+) ללא נפילת מתח.
- Pi GPIO 5V (ספק USB-C 3A) מספק את הלוגיקה של PCA9685 (50mA) בלי
  להיות מושפע מתנודות הסרוו.
- שני המקורות חולקים GND משותף — תנאי הכרחי לאותות I²C נכונים.
- LM2596 הוחלף ל־MB-102 רק אחרי שהתגלה שה־MB-102 (AMS1117, 700mA)
  לא מספיק — מתועד מלא ב־`problems/001-servo-power.md`.

### 8.2.4 USB UVC במקום CSI

ההחלפה ל־USB אחרי שמצלמת ה־Pi 5 לא תאמה את Pi 4 הפכה ליתרון:

- תאמה אוניברסלית — אפשר להחליף את ה־LifeCam בכל מצלמת USB אחרת
  בלי שינוי קוד.
- ה־OpenCV מתעלם מהמקור — `cv2.VideoCapture(0)` עובד עם USB וגם
  עם CSI עם ספריית bridge.
- כבל USB גמיש וארוך — מאפשר להציב את המצלמה במרחק 1m מה־Pi בלי
  בעיה.
- אין device tree config — פחות תחזוקה ב־OS.

### 8.2.5 MOSFET + Pulldown (קומבינציה בטיחותית)

זו לא בחירה של רכיב יחיד אלא של **טופולוגיית מעגל**:

- IRLZ44N נפתח ב־3.3V (GPIO לוגי) — אין צורך ב־level shifter.
- 220Ω בין GPIO ל־Gate מגן על ה־Pi משטף זרם בעת קצר.
- 100kΩ Pulldown מבטיח GPIO LOW בעת ה־boot — לייזר OFF כברירת
  מחדל.
- 100Ω בשורה עם הלייזר מגביל את זרם הדיודה ל־20mA — בטוח לדיודה
  5mW.

ארבעת הרכיבים יחד נותנים מערכת **fail-safe**: גם אם הקוד נכשל,
גם אם ה־Pi לא מאתחל, גם אם ה־GPIO צף — הלייזר נשאר OFF.

## 8.3 למה הספריות האלה — מבט תוכנתי

### 8.3.1 OpenCV + NumPy — תשתית עיבוד תמונה

הספריות האלה לא רק נבחרו; הן **קובעות את מבנה הקוד**:

- כל תמונה היא `numpy.ndarray` — אין צורך להמיר בין פורמטים.
- כל פונקציה ב־OpenCV ($cv2.cvtColor$, $cv2.inRange$, $cv2.erode$,
  $cv2.findContours$, $cv2.moments$) מקבלת ומחזירה ndarray.
- מקופלות ב־C++ — לכן `O(WH)` פעולות אורכות ms בודדות גם על Pi.
- מותקנות ב־apt (`python3-opencv`, `python3-numpy`) — אין צורך
  לקמפל ב־venv. החיסכון: שעות.

### 8.3.2 ServoKit + Blinka + gpiozero — שכבת הפשטה אחידה

שלוש הספריות האלה (ServoKit לסרוו, gpiozero לפין הלייזר, Blinka
כשכבת ההתאמה) חולקות פילוסופיה: **API גבוה, ניהול מצב פנימי,
טיפול בטוח ב־cleanup**. התוצאה — קוד הסרוו והלייזר נכתב בפחות
מ־50 שורות כל אחד, מה שמותיר את עיקר הקוד ללוגיקה (`tracker.py`,
`detector.py`).

### 8.3.3 simple-pid — כל מה שצריך, כלום מיותר

הספרייה היא פחות מ־400 שורות קוד. ה־API:

```python
pid = PID(Kp, Ki, Kd, setpoint=0, output_limits=(-LIM, LIM))
correction = pid(error)
```

מספק בדיוק את מה שדרוש: חישוב אוטומטי של dt, anti-windup על
`output_limits`, אפס תלות חיצונית. החלופה (כתיבה עצמית) הייתה
מסכנת באג עדין באנטי־וינדאפ או בחישוב dt.

### 8.3.4 tkinter — בלי תלות נוספת

הבחירה ב־tkinter (לעומת PyQt או web GUI) נובעת מעיקרון: **כל תלות
חדשה היא מקור פוטנציאלי לכשלי התקנה ב־Pi**. tkinter מוטמע ב־
Python — מובטח שיעבוד בכל סביבת Pi עם Python 3.11.

הסטאק התוכנתי הסופי דורש **6 ספריות חיצוניות** (`opencv-python`,
`numpy`, `gpiozero`, `adafruit-circuitpython-servokit`,
`adafruit-blinka`, `simple-pid`), שלוש מתוכן מותקנות ב־apt. זה
המינימום המעשי.

## 8.4 למה האלגוריתמים האלה — מבט אלגוריתמי

### 8.4.1 HSV Thresholding — מהירות ויציבות

הבחירה מבוססת על שתי תכונות של הבעיה:

- **המטרה ידועה מראש** (שקית כחולה ייחודית בתמונה) — אין צורך
  בזיהוי גנרי של "אובייקטים".
- **התאורה משתנה במהלך היום** — RGB יישבר; HSV יציב.

HSV נותן את ה־accuracy הנדרשת (זיהוי נקודתי 99%+) ב־$O(WH)$ — קטן
מסדר גודל מ־neural alternatives.

### 8.4.2 PID פרופורציוני בלבד — מינימליזם מכוון

ה־Kp = 0.017, Ki = 0, Kd = 0 הוא **תוצאה של ניסויים** (פירוט
ב־§14), לא הבחירה הראשונית:

- **למה לא Ki?** במערכת ללא חיכוך מתמיד (סרוו מתפצל בקלות), אין
  סטיית offset לסגור. Ki מוסיף latency בלי תועלת.
- **למה לא Kd?** הקלט (centroid מ־OpenCV) רעיש (~10px פריים־לפריים).
  Kd מגביר את הרעש לכדי תיקוני־יתר אגרסיביים.

הבחירה הסופית — פשוטה אך לא טריוויאלית — היא תוצאה של חקירה
אמפירית עם 8 גלגולי כיוון.

### 8.4.3 Coast + Recenter — חוסן בלי overkill

המנגנונים האלה מטפלים בשני סוגי כשל:

- **אובדן זמני (~1 שנייה)**: Coast מאפשר לברקט להמשיך בכיוון
  האחרון — דומה לאדם שעוקב במבט.
- **אובדן מתמשך**: Recenter מחזיר את הברקט למרכז כדי שיהיה מוכן
  למטרה הבאה.

החלופה (Kalman Filter) הייתה נותנת ~10% שיפור באיכות החיזוי, אבל
עם 50 שורות קוד נוספות ושלושה פרמטרים חדשים. ההחלטה: KISS עכשיו,
Kalman עתידי אם יוצדק (§16).

### 8.4.4 מכונת מצבים סופית — שליטה במורכבות

חמשת המצבים (Tracking, Locked, Coasting, Recentering, Holding)
ממסגרים את כל המקרים. כל מעבר ביניהם מתועד בקוד ובעריכה
ב־`tracker.py`. בלי המבנה הזה, הקוד היה הופך ל"כדור spaghetti של
if-else" שקשה לדבג.

## 8.6 סינרגיות — איפה הסטאק > סכום החלקים

חלק מהבחירות עובדות יחד באופן שמייצר יתרונות שלא היו קיימים בכל
חלופה לחוד:

1. **Bookworm 64-bit + apt OpenCV + Python 3.11 + venv --system-site-packages**
   — מאפשר התקנת ספריות PyPI חדשות (servokit, simple-pid) בלי
   לאבד את ה־OpenCV של המערכת. הצירוף הזה חוסך שעות של הקמת
   סביבה.

2. **PCA9685 + ServoKit + 16 ערוצים זמינים** — הוספת סרוו שלישי
   (לדוגמה: זרוע לחיצה לכפתור פיזי) דורשת רק חיווט וערכת שורה
   בקוד `servo.py`. ה־PWM driver, ה־I²C, ההפשטה — כבר במקום.

3. **MOSFET + Pulldown + clamping תכנותי + try/finally** — ארבע
   שכבות בטיחות עצמאיות. אם אחת נכשלת, השאר ממשיכות לפעול. כל
   אחת לבדה לא מספיקה; ההגנה היא ה־*intersection* של ארבעתן.

4. **Owner Modules + SSOT + apt OpenCV + Git auto-pull** — מאפשר
   workflow פיתוח שלם של "עורך בלפטופ, רואה תוצאה ב־Pi בתוך
   דקה". כל אחת מהבחירות לבדה לא הייתה מספיקה.

5. **HSV + Deadband + Coast + Recenter** — הצירוף הזה מאפשר
   ל־P-only PID (פשוט, יציב) לעמוד בכל המצבים. בלי deadband, היה
   רטט; בלי Coast, מטרות מהירות היו אובדות; בלי Recenter, הברקט
   היה נתקע בפינות.

## 8.7 סיכונים ידועים בסטאק

הבחירה היא טובה אך לא מושלמת. הסיכונים העיקריים שזוהו מראש:

| סיכון | סבירות | חומרה | הקלה |
|---|:---:|:---:|---|
| GIL של Python מגביל ביצועים אם הוסיפו threads | בינונית | בינונית | במצב הנוכחי הלולאה רצה ב־30 fps עם פנאי; אם יידרש — מעבר ל־multiprocessing |

| OpenCV apt לא תמיד עדכני | נמוכה | נמוכה | אם דרושה גרסה ספציפית, אפשר לקמפל. עד היום לא היה צורך |
| Bookworm 64-bit עדיין יחסית חדש | נמוכה | נמוכה | יש fallback ל־32-bit; אך 64-bit מומלץ רשמית |
| הסרוו יישחק בשימוש מתמשך | בינונית | נמוכה | DS3225 מסוגלים ל־1M+ מחזורים; פרויקט שולחני לא יגיע אליהם |

## 8.8 סיכום

הסטאק שנבחר הוא **פשרה מודעת** בין:

- **ביצועים** (30 fps ב־Pi 4B) — לא מקסימליים אבל מספיקים.
- **עלות** (~$80 כולל הכל) — בקצה הנמוך של הפתרונות המעשיים.
- **גמישות** (Owner Modules + SSOT + 14 ערוצי PCA9685 פנויים) —
  מאפשר שדרוגים בלי לשבור את התשתית.
- **בטיחות** (4 שכבות עצמאיות) — overkill לפרויקט סטודנט, אבל מהווה
  הרגל טוב לפרויקטים גדולים יותר.
- **למידות** (קוד פתוח, ספריות בשלות, תיעוד מקיף) — מתאים להגשה
  לפיקוח ולשימוש כבסיס לפרויקטי המשך.

אף בחירה אחת לא הייתה דרמטית לבדה. ה־*קומבינציה* היא מה שהופך את
הסטאק למתאים — כל בחירה מאפשרת את השאר.
