# 11. תיאור הארכיטקטורה

הפרק מתאר את הארכיטקטורה של המערכת בשתי רמות: ארכיטקטורה חומרתית (איזה רכיב מתחבר לאיזה, מאיפה זורם מה לאן), וארכיטקטורה תוכנתית (איזה קוד מדבר עם איזה, מי בעלי של מה).

## ארכיטקטורה חומרתית

המערכת מורכבת משלוש "שכבות": שכבת ה**ספק**, שכבת ה**עיבוד**, ושכבת ה**אקטואטור והחיישנים**.

**שכבת הספק** מפיצה חשמל בכל המערכת ממקור אחד או יותר. מקור 1 הוא **ספק קיר 12V/5A** שמזין את ה־**LM2596 buck converter**, שמוריד את המתח ל־5V יציב לסרוואים ושופך אותו לפין V+ של ה־PCA9685 (3A מקסימום). מקור 2 הוא **ספק USB-C 5V/3A** שמזין את ה־Pi עצמו, שמספק 5V מפין 2 של ה־GPIO header שלו ל־VCC של ה־PCA9685 (לוגיקה בלבד). שתי המסילות נפרדות לחלוטין מבחינת חשמל — אבל ה־GND שלהן מחוברת כדי שהאותות יוכלו לחצות ביניהן ללא רעש.

**שכבת העיבוד** היא ה־**Raspberry Pi 4B** שיושב במרכז. הוא לא מקבל שום קלט אנליוגי או חשמלי מהמערכת — הקלט שלו הוא וידאו דיגיטלי דרך USB מהמצלמה. הפלט שלו הוא פקודות I²C שיוצאות מ־GPIO2/3 ל־PCA9685, ופקודת HIGH/LOW על GPIO18 שיוצאת אל הגייט של ה־MOSFET של הלייזר.

**שכבת האקטואטור והחיישנים** היא הקצוות של המערכת. החיישן היחיד הוא **המצלמה** — LifeCam HD-3000 על USB. האקטואטורים הם **שני סרוואי DS3225** (ערוצים 0 ו־1 של PCA9685) ו**הלייזר** (תחת בקרת MOSFET שמוזן מ־5V דרך נגד מגבל זרם).

תרשים הקישור:

```
   12V/5A PSU                        USB-C 5V/3A
       │                                  │
       ▼                                  ▼
   LM2596 (5.0V)                  Raspberry Pi 4B
       │                            │ │ │ │ │
       │ V+                      P2 │ │ │ │ │ GPIO18
       ▼                       (5V) │ │ │ │ │   │
   ┌─────────┐                      │ │ │ │ │   ▼
   │PCA9685  │←─────────────────────┘ │ │ │ │  220Ω
   │ 0x40    │←(SDA)─────GPIO2────────┘ │ │ │   │
   │         │←(SCL)─────GPIO3──────────┘ │ │   ▼
   │         │←(GND)─────GPIO6────────────┘ │ Gate
   └────┬────┘                              │   │
        │ channels 0,1                      │   │
        │                                   │  IRLZ44N
        ▼                                   │  ┌──┐
   ┌─────────┐                              │  └──┘
   │DS3225 ×2│                              │   │ Drain
   │ pan,tilt│                              │   │
   └─────────┘                              │   │
                                            │  Laser
                                            │   │
                                            │   ▼
                                            │  100Ω
                                            │   │
                                            └──5V (P4)
```

ה־GND משותף לכל הרכיבים — קיבעון יחיד של "אדמה" שמאפשר השוואת מתחים בין מסילות שונות בלי טעויות. כל מקרה של כשל אספקה (Pi מאבד מתח, סרוו לא מקבל מספיק) נדון ב־[`problems/`](../../problems/).

## ארכיטקטורת תוכנה

הקוד מאורגן לפי **דפוס המודולים הבעלים**. הרעיון: כל תת־מערכת חומרה (סרוו, מצלמה, לייזר) נמצאת בבעלות בלעדית של קובץ Python אחד, והקובץ הזה הוא היחיד שמורשה לדבר עם הספרייה החיצונית של אותה חומרה. כך השאר של הקוד לא יודע, ולא צריך לדעת, איזה ספרייה ספציפית עומדת מאחורי כל פעולה.

מפת הבעלות:

| מודול | בעלים על | מורשה לייבא |
|---|---|---|
| [`servo.py`](../../servo.py) | PCA9685 + סרוואי DS3225 | `adafruit_servokit` |
| [`camera.py`](../../camera.py) | מצלמת USB | `cv2.VideoCapture` |
| [`detector.py`](../../detector.py) | אלגוריתם זיהוי HSV | `cv2` (לעיבוד תמונה, לא ל־capture) |
| [`tracker.py`](../../tracker.py) | בקר PID | `simple_pid` |
| [`laser.py`](../../laser.py) | GPIO18 + הלייזר | `gpiozero.LED` עבור GPIO18 |
| [`config.py`](../../config.py) | קבועים מכוילים | `numpy` |
| [`control_panel.py`](../../control_panel.py) | GUI tkinter | מודולי הפרויקט (servo, camera, etc.) |

הציות לדפוס נאכף קונבנציונית: אנחנו לא משתמשים ב־`import hooks` או בדיקות אוטומטיות כדי לאסור על קוד לייבא `gpiozero` במקום אחר חוץ מ־`laser.py`. במקום זה, [`CLAUDE.md`](../../CLAUDE.md) מתעד את הדפוס במפורש, ובכל session של פיתוח הוא הקובץ הראשון שנקרא. בפועל זה עבד: על פני 8 שלבים של פיתוח אף קוד אחר לא ייבא את הספריות החומרתיות.

מעל המודולים הבעלים יושבים **סקריפטי הפעלה** שמרכיבים אותם יחד:

- [`test_tracking.py`](../../test_tracking.py) — מריץ את לולאת המעקב המלאה. פותח camera, יוצר detector, מאתחל tracker עם שני בקרי PID, נכנס ל־loop של 30Hz: capture → detect → update PID → move servos. מציג overlay של הסטטוס (pan/tilt error, סטטוס Tracking/Locked/Coasting/Recentering). יוצא ב־`q`.
- [`tune_detector.py`](../../tune_detector.py) — GUI כיול HSV. סליידרים לבחירת H/S/V min/max, חלון חי שמציג את המסכה הבינארית. שומר ל־`config.py` בלחיצת כפתור.
- [`calibrate_servo.py`](../../calibrate_servo.py) — סקריפט אינטראקטיבי לקביעת הגבולות הבטוחים של פאן וטילט. שואל את המפעיל לפי הסדר: "האם הברקט נע חופשי?", "האם יש קליק/חריקה?", ושומר את הערכים שהמפעיל אישר.
- [`test_servo.py`](../../test_servo.py) — סקריפט סניטיציה לבדיקה ש־PCA9685 ו־DS3225 מגיבים תקין. מזיז כל סרוו ב־5° צעדים.
- [`test_laser.py`](../../test_laser.py) — בדיקת לייזר עצמאית. countdown של 3 שניות (זמן להסתכל הצידה), פולס של שנייה, כיבוי. בודק שה־MOSFET, ה־GPIO וה־pulldown עובדים.
- [`main.py`](../../main.py) — תיכלול סופי של כל הצינור: tracker → camera → detector → servo → laser. ה־loop הראשי של המערכת בפעולה רגילה.

ה־**`control_panel.py`** הוא ה־"shell" של כל אלה: לחיצת כפתור ב־GUI מפעילה את הסקריפט הרלוונטי כ־subprocess. הסיבה שזה subprocess ולא קריאה ישירה לפונקציה היא ש־`test_tracking.py` ו־`tune_detector.py` פותחים חלונות של cv2, ו־cv2 לא יודע לשתף mainloop עם tkinter — שניהם דורשים שליטה על ה־event loop של ה־thread הראשי. הפרדה ל־processes פותרת את ההתנגשות בלי שום קוד מיוחד.

## איך השכבות מדברות

הכיוון של המידע הוא: **חומרה → driver → owner module → סקריפט הפעלה → GUI**.

בקריאת פריים, למשל: המצלמה מפיקה תמונה ב־480×640 BGR, הקרנל של Linux קורא אותה דרך `uvcvideo` לבאפר בזיכרון, OpenCV (`cv2.VideoCapture.read()`) מעבירה את הבאפר למערך numpy, `camera.capture_frame()` מחזירה את המערך, `detector.detect()` מעבדת אותו וחישבת `(cx, cy)`, `tracker.update()` מחשבת מזה תיקון של זוויות, `servo.move_pan/move_tilt` שולחים פקודה ל־PCA9685, ה־PCA9685 מעדכן את ה־duty cycle, הסרוו זז. בכיוון השני: **GUI → לוגיקה → סקריפט → owner module → driver → חומרה**. לחיצה על כפתור Fire ב־control panel קוראת לפונקציה ב־handler, שקוראת ל־`laser.fire(laser_dev)`, שקוראת ל־`laser_dev.on()` של gpiozero, שכותבת `1` ל־`/sys/class/gpio/gpio18/value`, שמרים את ה־GPIO ל־HIGH, שפותח את ה־MOSFET, שמפעיל את הלייזר.

הפרדת השכבות הזו היא מה שמאפשר את היכולת להחליף רכיב בלי לשבור את שאר המערכת. במהלך הפיתוח החלפנו פעמיים את הספק (MB-102 ל־LM2596 — לא נגענו בקוד), פעם את המצלמה (CSI ל־USB — שינוי רק בקובץ camera.py), ופעמיים את ערכי הכיול (HSV, PID — שינוי רק ב־config.py).
